# 后续学习路径优化

这个文件用于把 Python100 调整成更适合“已经学到 Day10、开始真实敲代码、但终端/路径/文件操作还不稳定”的学习者。

目标不是学得更花。

目标是学得更稳、更能迁移、更能形成自己的小作品。

## 当前学习画像

已经具备：

- 会在终端里创建目录、进入目录、运行 Python 文件。
- 会用 `nano` 或编辑器修改代码。
- 会写基础变量、循环、条件判断和函数。
- 已经接触文件读写、异常处理、`main()` 入口。
- 能把一个数学规律写成多种 Python 解法。

当前短板：

- 路径、文件名、目录位置还容易混乱。
- 空文件、拼写错误文件名、运行错文件的情况会出现。
- Debug 还偏靠反复试，不够系统。
- 代码能跑，但结构化说明、测试和复盘还弱。

因此后续学习要多做“可运行小任务”，少做纯文字阅读。

## Day11-Day20 优化重点

这一阶段不要急着做大工程。

核心任务是把基础变稳。

每天必须完成：

- 一个最小可运行脚本。
- 一次故意制造错误再修复。
- 一次文件路径检查。
- 一次函数拆分。
- 一次结果复盘。

重点练：

- CSV / JSON 的读写。
- 字符串清洗。
- 列表、字典、集合的组合使用。
- `break`、`continue`、`return` 的区别。
- `try/except` 的边界。
- `main()` 入口和命令行运行。

判断是否过关：

- 能解释当前文件在哪里。
- 能解释输入文件是什么。
- 能解释输出文件是什么。
- 能解释出错时先查哪三件事。

## Day21-Day35 优化重点


## 创新思维训练

这里的创新不是做很大的东西。

而是训练四种小创新：

你已经出现过一个很重要的苗头：

```text
从统计里的假设检验、吸附壁、示性函数出发，
把抽象判断转成 Python 里的条件、符号、权重和算法优化。
```

这类思路要继续保留。

它说明你不是只在背语法，而是在尝试把数学/统计直觉翻译成代码。

### 1. 一题多解

同一个问题至少想两种写法。

例如：

- `if/else` 写法。
- 数学公式写法。
- 函数封装写法。
- 字典映射写法。

这能训练“我不是只会照抄一种答案”。

例如 Day10 的示性函数思路：

```python
def sign_odd_even(x):
    if x % 2 == 1:
        return 1
    return -1
```

它表面是在算正负号。

更深一层是在训练：

- 把判断条件抽象成函数。
- 把 `if/else` 变成可复用规则。
- 把数学里的示性函数变成程序里的权重函数。
- 后续可以迁移到 Quant 信号、分类规则、异常标记、Agent 路由。

再进一步，可以优化成公式：

```python
sign = 2 * (x % 2 == 1) - 1
```

这就是从“能写出来”到“能优化表达”的过程。

### 2. 场景迁移

每学一个语法，都问它能迁移到哪里。

例如：

- `break`：遇到严重错误立刻停止任务。
- `continue`：跳过坏数据，继续处理下一行。
- 字典：做配置、计数、索引、缓存。
- JSON：做 Agent tool 的输入输出。

统计背景可以这样迁移：

- 假设检验：转成“先设原假设，再用数据证据判断是否拒绝”的分析流程。
- 示性函数：转成 `0/1` 标签、过滤条件、权重函数、交易信号。
- 吸附壁/阈值直觉：转成边界条件、停止条件、风险阈值、异常拦截。
- 显著性：转成结果是否足够可靠、是否需要更多样本、是否能写进报告。

### 3. 小工具意识

每 10 天做一个很小的工具。

工具不需要大。

但要真的解决一个小问题。

例如：

- 学习记录 CLI。
- CSV 体检器。
- 文件整理器。
- Prompt 输出检查器。
- 回测结果保存器。

### 4. 错误改造

把错误变成材料。

每次路径错、文件名错、运行错文件，都记录成一条：

```text
错误：
原因：
修复：
以后怎么避免：
```

这会形成真正的 Debug 能力。

## 创新思维落地方法

你的优势不是一开始就写大系统。

你的优势更像是：

```text
从一个统计/数学直觉出发，
把它变成规则，
再把规则变成函数，
最后把函数放进数据流或工具里。
```

后面每遇到一个概念，都用这个 5 步法：

```text
1. 这个概念在判断什么？
2. 判断需要哪些输入？
3. 输出是 0/1、分数、标签，还是一段文本？
4. 能不能写成一个函数？
5. 能不能放进 CSV/JSON/pandas/Agent/Quant 流程？
```

### 假设检验式思维

写代码时可以这样迁移：

```text
原假设：当前数据没有异常。
备择假设：当前数据存在异常。
证据：缺失率、波动率、阈值、错误次数。
决策：通过 / 拒绝 / 需要人工检查。
```

对应代码结构：

```python
def check_anomaly(value, threshold):
    if value > threshold:
        return "reject"
    return "pass"
```

后续可迁移到：

- 数据质量检查。
- Quant 风险阈值。
- LLM 输出是否可信。
- Agent 是否需要人工审查。

### 吸附壁/阈值思维

你说的“吸附壁”可以理解成边界或阈值。

在代码里，它常常变成：

- 上限。
- 下限。
- 停止条件。
- 风险边界。
- 自动拦截线。

示例：

```python
def clamp_score(score):
    if score < 0:
        return 0
    if score > 100:
        return 100
    return score
```

后续可迁移到：

- 数据清洗里的异常截断。
- Quant 仓位上限。
- Agent 工具调用次数上限。
- LLM 输出置信度阈值。

### 示性函数思维

示性函数就是把条件变成 `0/1` 或 `-1/1`。

它非常适合你后面的路线。

```python
def indicator(x, threshold):
    return int(x > threshold)
```

可以迁移到：

- 是否缺失。
- 是否异常。
- 是否触发交易信号。
- 是否需要人工审核。
- 是否满足 prompt 输出格式。

### 算法优化思维

优化不是一上来追高级写法。

按这个顺序：

```text
先写能看懂的版本。
再写少重复的版本。
再写函数封装版本。
最后才考虑性能。
```

例如：

```text
if/else 版本 -> 函数版本 -> 公式版本 -> 向量化版本
```

这条路线很适合你。

因为你已经会从示性函数想到算法表达。
