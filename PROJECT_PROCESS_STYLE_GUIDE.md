# 项目过程化与聊天式学习风格指南

这份指南是 Python100 后续重构的最高优先级之一。

它回答一个现实问题：

> 这些项目做出来真的有用吗？

不能保证。

任何课程都不能保证一个预设项目天然有用。真正有长期价值的，不是“做了某个项目名字”，而是学会一套可迁移的做项目过程。

所以本课程的重心要从：

```text
做一个项目
```

改成：

```text
学习如何把一个模糊想法做成项目
```

项目只是练习场，不是承诺。

## 新核心目标

Python100 不再把“数据工具、LLM 助手、Agent 原型、Quant Starter、本地 RAG”当成最终保证有用的作品。

它们只是五类训练场景。

真正要训练的是：

1. 如何发现一个小问题。
2. 如何把问题压成输入和输出。
3. 如何定义最小可运行版本。
4. 如何先写手工流程，再写代码。
5. 如何处理失败、脏数据、空输入和边界。
6. 如何加日志、测试、README 和样例数据。
7. 如何复盘这个项目是否值得继续。
8. 如何决定停止、缩小、重做或迁移。

## 项目过程八步

每个“项目型章节”都应该围绕这八步讲，而不是围绕项目名炫技。

### 1. 问题从哪里来

先问：

- 我为什么要做这个东西？
- 它解决我哪个重复麻烦？
- 如果不做，会损失什么？
- 这个问题是不是足够小？

不要一开始就问“用什么框架”。

### 2. 谁会用

哪怕只有自己用，也要说清楚：

- 使用者是谁？
- 他在什么场景下打开它？
- 他最想少做哪一步？
- 他能接受命令行，还是需要网页？

### 3. 输入是什么

输入一定要具体。

例如：

- 一行学习记录。
- 一个 CSV 文件。
- 一段日志文本。
- 一组行情样例。
- 一个用户问题。
- 一批 Markdown 笔记。

模糊输入会导致项目失控。

### 4. 输出是什么

输出也要具体。

例如：

- 终端一句提示。
- 一个 CSV 文件。
- 一个 JSON 对象。
- 一张图。
- 一份 Markdown 报告。
- 一个 API 响应。
- 一个 README 示例。

如果输出说不清，项目还没有准备好写代码。

### 5. 最小版本是什么

最小版本不是低级。

最小版本是为了让你快速验证：

- 这个流程能不能跑通？
- 输入输出是否真的有意义？
- 哪一步最容易失败？
- 有没有继续做的必要？

最小版本应该小到 1-2 小时能完成。

### 6. 失败点在哪里

每个项目都要提前问：

- 空输入怎么办？
- 文件不存在怎么办？
- 字段缺失怎么办？
- 类型不对怎么办？
- 网络失败怎么办？
- 模型回答乱了怎么办？
- 结果看起来合理但其实错了怎么办？

这一步比加功能更重要。

### 7. 如何交付

交付不是上线。

入门阶段的交付可以很轻：

- 一个可运行脚本。
- 一个样例输入。
- 一个样例输出。
- 一个 README。
- 一个测试。
- 一段 Debug 记录。
- 一段复盘。

### 8. 是否值得继续

做完最小版本后，要敢于判断：

- 继续扩展。
- 暂停。
- 缩小。
- 合并进别的项目。
- 只保留为代码片段。
- 删除复杂想法，只留学习结论。

这才是长期主义。

## 聊天式学习风格

课程语气要接近一个好的 GPT 对话。

不是论文。

不是百科全书一次倒完。

而是：

```text
短句。
小段。
一步一个问题。
先说人话。
再给代码。
再解释为什么。
最后给下一步。
```

## 单次短，总体长

新的写作原则：

- 单段尽量短。
- 一段只讲一个意思。
- 每次只推进一个小台阶。
- 总体内容可以很长。
- 但读者每次只需要消化一小块。

也就是说：

```text
不是减少内容。
而是降低单次吞咽难度。
```

## 每日推荐结构

每一天最好改成这种节奏：

```text
1. 今天先别急着写代码
2. 你现在遇到的真实问题是什么
3. 这个问题的输入是什么
4. 这个问题的输出是什么
5. 最小版本只做哪一步
6. 需要哪个 Python 知识点
7. 先手写流程
8. 再写最小代码
9. 看它哪里会失败
10. 加一个边界处理
11. 记录 Debug
12. 今天先停在这里
```

这种结构比“定义、语法、例题”更适合长期学习。

## 项目名降级规则

原来的项目名仍然可以保留，但要降级为例子。

例如：

```text
QuantResearchStarter
```

不要讲成：

```text
我们要做一个很完整的 Quant 项目。
```

要讲成：

```text
我们用 QuantResearchStarter 练习一个过程：
如何把一个策略想法拆成数据、指标、信号、回测、报告和复盘。
```

再例如：

```text
LLMReportAssistant
```

不要讲成：

```text
我们要做一个 LLM 报告助手。
```

要讲成：

```text
我们用 LLMReportAssistant 练习一个过程：
如何把一段材料变成有上下文、有事实边界、有结构化输出、有审查记录的报告流程。
```

## 课程承诺

本课程不承诺：

- 做完项目就能直接就业。
- 项目本身一定有商业价值。
- 每个项目都值得继续维护。
- 所有技术都会深入到生产级。

本课程承诺训练：

- 如何拆问题。
- 如何定义输入输出。
- 如何写最小版本。
- 如何处理失败。
- 如何复盘。
- 如何把代码沉淀成长期资产。

这比某个项目名字更可靠。
