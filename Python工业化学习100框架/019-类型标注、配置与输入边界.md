## 基础详细讲解

### 如果你完全看不懂，先只抓这一句话

今天学的是：**类型标注让函数输入输出更清楚，配置让变化的参数集中管理。**

```python
def average(values: list[float]) -> float:
    return sum(values) / len(values)
```

这句话不是让 Python 自动变聪明，而是告诉读者：`values` 应该是一组浮点数，返回值应该是浮点数。

### 类型标注怎么读

| 写法 | 含义 |
| --- | --- |
| `x: int` | x 应该是整数 |
| `name: str` | name 应该是字符串 |
| `values: list[float]` | values 应该是浮点数列表 |
| `-> float` | 函数返回浮点数 |

### 配置是什么

不要把经常变化的参数散落在代码里：

```python
threshold = 0.05
input_path = "data.csv"
```

更好的做法是集中：

```python
config = {
    "threshold": 0.05,
    "input_path": "data.csv",
}
```

以后要改，就知道去哪里找。

### 边界设计怎么想

函数入口先问：

```text
这个输入允许为空吗？
类型对吗？
范围合理吗？
缺字段怎么办？
```

这就是后面 API、Agent tool、Quant 参数实验的基础。

### 类型标注、配置与输入边界

类型标注不是给机器看的装饰，而是给未来读代码的人看的契约。配置把经常变化的参数从代码里拿出来，输入边界则说明什么数据可以进入系统。

### 基础知识深讲：类型标注、配置和边界设计

类型标注是一种可读契约。`def average(values: list[float]) -> float` 告诉读者：这个函数希望接收浮点数列表，并返回浮点数。它不等于运行时自动检查，但能让代码意图更清楚，也方便工具发现一部分问题。

配置是把可变参数从代码逻辑里分离出来。路径、阈值、开关、模型名、样本区间都可能变化；如果它们散落在代码里，修改和复查都会困难。配置文件让“业务参数”和“处理逻辑”分开。

输入边界设计决定系统稳定性。入口处应该检查必要字段、类型、范围和空值；核心函数则尽量处理已经验证过的数据。这样代码结构会更清楚：入口负责防守，核心负责计算。

这一天是从语法转向工程判断的关键。你不只是写能跑的代码，而是在定义别人如何安全地调用你的函数、脚本或工具。后面 Agent tool schema、API request validation、Quant 参数实验都延续这个思路。

### 手写顺序

- 给函数参数和返回值加简单类型，例如 `list[float] -> float`。
- 把路径、阈值、开关放到配置里，不要散落在代码中。
- 在入口处检查输入，在核心函数里保持逻辑干净。

### 今天必须真正弄懂

- **输入**：今天这段代码从哪里拿到数据，是手写值、终端输入、文件，还是函数参数。
- **处理**：每一行代码改变了什么状态，例如变量变了、列表多了一个元素、判断走了某个分支。
- **输出**：结果给谁看，是终端输出、返回值、文件，还是后面函数继续使用。

### 零基础拆解：类型、配置和边界

#### 这一天到底在练什么

类型标注让输入输出更清楚，配置让可变参数集中管理，边界检查让坏输入在入口处被发现。

#### 手写时按这个顺序

- 先给一个函数加参数和返回值类型。
- 把路径或阈值写进 config。
- 在入口处检查必要字段是否存在。

#### 最常见卡点

类型标注不会自动修复逻辑，它主要帮助阅读和工具检查。真正的边界仍然要靠判断代码。

- 我能读懂 `list[float] -> float`。
- 我能把阈值从代码移到配置。
