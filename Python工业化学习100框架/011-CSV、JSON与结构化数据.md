### 最小案例

```python
import csv
import json

rows = [{"topic": "python", "minutes": 120}, {"topic": "sql", "minutes": 90}]
with open("learning.json", "w", encoding="utf-8") as f:
    json.dump(rows, f, ensure_ascii=False, indent=2)
with open("learning.csv", "w", encoding="utf-8", newline="") as f:
    writer = csv.DictWriter(f, fieldnames=["topic", "minutes"])
    writer.writeheader()
    writer.writerows(rows)
```

## 基础详细讲解

### 如果你完全看不懂，先只抓这一句话

今天学的是：**CSV 像表格，JSON 像字典和列表的组合。读取后先检查结构，再分析。**

```text
CSV -> 一行一条记录，一列一个字段
JSON -> dict / list 嵌套结构
```

### CSV 怎么想

CSV：

```text
topic,minutes
python,120
sql,90
```

可以想成：

```python
[
    {"topic": "python", "minutes": "120"},
    {"topic": "sql", "minutes": "90"},
]
```

注意：从文件读出来的 `"120"` 很可能先是字符串，要计算就要转成数字。

### JSON 怎么想

```json
{"topic": "python", "minutes": 120}
```

读进 Python 后很像：

```python
{"topic": "python", "minutes": 120}
```

所以 JSON 和 dict 的关系非常近。

### 读取后先做三件事

| 检查 | 为什么 |
| --- | --- |
| 打印类型 | 确认是 list 还是 dict |
| 打印数量 | 确认读到了多少条 |
| 打印第一条 | 确认字段名和数据形状 |

### Debug 对照表

| 现象 | 常见原因 | 怎么改 |
| --- | --- | --- |
| 字段取不到 | 字段名拼错或表头不同 | 打印所有 key |
| 数字不能相加 | 读出来是字符串 | 用 `int()` 或 `float()` 转换 |
| JSON 报错 | 格式不是合法 JSON | 检查引号、逗号、括号 |
| 循环层级错 | 把 dict 当 list 或把 list 当 dict | 先打印 `type(data)` |

### CSV、JSON 与结构化数据

CSV 像表格，JSON 像嵌套字典和列表。学习它们的重点不是记格式，而是知道数据从哪里来、字段是什么意思、读取后变成什么 Python 对象。

### 基础知识深讲：结构化数据和字段意识

CSV 和 JSON 的共同点是把数据结构显式保存下来。CSV 更像二维表，每一行是一条记录，每一列是一个字段；JSON 更像 Python 的 dict/list，可以表达嵌套结构。理解格式差异，才能决定后面用什么方式读取和验证。

读取结构化数据后，不要马上计算。第一步应该做数据体检：顶层类型是什么、总共有多少条、字段名有哪些、第一条样例长什么样。这个流程能提前发现字段拼写、缺失值、类型不一致和格式误读。

字段意识是数据分析的基础。`price`、`date`、`volume`、`topic` 都不是普通变量名，它们代表业务含义。代码里每次取字段，都应该能说清这个字段从哪里来、单位是什么、是否允许为空。

JSON 尤其要注意层级。`data["items"][0]["title"]` 这种路径表示从字典进入列表，再进入字典。以后调用 LLM API、读取配置、处理 Agent tool 输出时，嵌套结构会非常常见。

### 手写顺序

- CSV 先看表头，再逐行读取；字段名要和代码里的 key 对得上。
- JSON 先打印顶层类型：是 dict 还是 list，再逐层取字段。
- 读完数据后先做数量、字段、样例三项检查。

### 今天必须真正弄懂

- **输入**：今天这段代码从哪里拿到数据，是手写值、终端输入、文件，还是函数参数。
- **处理**：每一行代码改变了什么状态，例如变量变了、列表多了一个元素、判断走了某个分支。
- **输出**：结果给谁看，是终端输出、返回值、文件，还是后面函数继续使用。

#### 这一天到底在练什么

CSV 像表格，JSON 像 Python 里的 dict/list 组合。今天重点是读进来之后先检查结构，而不是马上分析。

## 工程化详细讲解

### 如果你完全看不懂，先只抓这一句话

今天这章不是孤立知识点，而是在训练一个真实工程习惯：把「CSV、JSON与结构化数据」放进一个可以运行、可以检查、可以复用的小流程里。

这一阶段的目标是把基础语法变成可复用的小能力。你要从“我会写一段代码”，过渡到“我知道这段代码放在哪里、输入是什么、输出是什么、失败时怎么查”。

### 真实工作流怎么想

你可以按下面这条线理解今天内容：

```text
输入数据 -> 选择数据结构 -> 写成函数 -> 打印/保存结果 -> 记录错误
```

这条线的意思是：先不要追求写得高级。先保证你知道数据从哪里来，经过了哪些变量，最后去了哪里。只要能画出这条数据流，你就已经从“背语法”进入“写程序”。

### 执行过程追踪

学习今天内容时，每写 5 到 10 行代码，就问自己四个问题：

1. 现在有哪些变量？
2. 每个变量是什么类型？
3. 下一行代码会改变哪个变量？
4. 如果输出不对，我应该打印哪里？

这四个问题比多背几个函数更重要。以后做数据分析、Agent tool、量化回测时，真正救命的就是这种追踪能力。

### 最容易混的点

- 不要只看最终答案，要看中间变量怎么变化。
- 不要把函数、循环、文件、列表当成分开的知识，它们最后都会组合在一个任务里。
- 不要一开始就追求“优雅写法”。先写清楚，再减少重复。
- 每个例子都要能说清楚 input、process、output。
