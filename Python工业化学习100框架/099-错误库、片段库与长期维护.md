## 工程问题解决结构

### 1. 是什么（What）

错误库、片段库与长期维护 是用 Python 解决工程任务时的一组小工具。最简定义：它帮助你把「整理 Debug 和 reusable snippets、maintenance、输入输出边界」放进可运行流程里。先把它当成解决问题的方法，不要先背术语。

### 2. 为什么存在（Why）

如果没有 错误库、片段库与长期维护，代码通常会变成一次性脚本：逻辑散、结果难复查、报错难定位、以后项目里也难复用。它存在的意义是把真实任务拆成更清楚的步骤，让你知道哪里输入、哪里处理、哪里输出。

### 3. 真实工程场景（Real Scenario）

- 科研场景：把阶段成果整理成可以公开展示和长期维护的研究/工程资产。
- 数据分析场景：把多个小工具组合成作品集项目，并写清输入输出。
- Quant 场景：把回测、报告、配置和 README 打包成可展示项目。
- LLM/Agent 场景：把 Agent/RAG/Tool 项目做成端到端演示，并保留日志和边界说明。

## 工程工作流补充

### 如果你完全看不懂，先只抓这一句话

今天学的是：**错误库、片段库与长期维护**。它不只是一个孤立知识点，而是服务于「项目实战与作品集」阶段的一个小能力：把前面的能力整理成可展示、可运行、可讲清楚的项目。

先按这条工程线理解：

```text
项目目标 -> 模块拆分 -> 样例输入 -> 测试/演示 -> README/复盘
```

你不需要一开始把所有细节背下来。先回答五个问题：

1. 今天的输入是什么？
2. 中间处理分几步？
3. 输出保存在哪里？
4. 失败时会在哪一步失败？
5. 这个能力以后能放进哪个项目？

### 真实工作流怎么想

把今天的主题放进一个真实项目里，它通常不是单独出现的。比如做一个 AI/data/quant 小工具时，你会经历：

| 步骤 | 你要做什么 |
| --- | --- |
| 1. 准备输入 | 明确数据、参数、文件、用户问题或样例记录从哪里来 |
| 2. 写核心函数 | 把最重要的一步处理封装成函数，先保证可运行 |
| 3. 检查边界 | 检查空输入、缺字段、类型不对、路径不对、网络失败等情况 |
| 4. 保存结果 | 把输出保存成 Markdown、JSON、CSV、SQLite、日志或网页 |
| 5. 写复盘 | 在 README 或学习日志里写清运行命令、输入输出和下一步 |

### 执行过程追踪

读今天代码时，不要只看语法，要追踪数据怎么走：

```text
输入
  ↓
核心处理
  ↓
中间结果
  ↓
边界检查
  ↓
最终输出
```

当前最应该检查：运行命令、目录结构、测试结果、演示输出、项目故事。

### 最容易混的点

| 卡点 | 正确理解 |
| --- | --- |
| 只跑通一次就以为学会了 | 至少换 2-3 组样例输入，确认逻辑可复用 |
| 只在终端 print，不保存结果 | 工程项目要留下可复查产物 |
| 报错后直接改代码 | 先记录触发输入、报错行、失败原因，再修复 |
| 函数越写越长 | 先拆成输入读取、核心处理、结果保存三个部分 |
| README 最后才写 | 先写运行命令，代码按运行命令实现 |

### 工程使用场景

今天的主题是 **错误库、片段库与长期维护**，不要只把它当成一个孤立知识点。它应该落到 `AI Builder 作品集` 里的一个可复用能力：输入清楚、处理过程可检查、输出可以保存，失败时能解释原因。重点检查：项目 README、测试、演示脚本、复盘和作品索引。

### 强化 7 题

1. **复现题**：不看答案重写今天最小案例，并写出运行命令和预期输出。
2. **边界题**：构造 3 个异常输入，说明程序应该返回结果、跳过、还是报错。
3. **数据题**：准备 5-10 条虚拟样例数据，覆盖正常值、空值、重复值和极端值。
4. **函数题**：把核心逻辑封装成 1 个函数，函数名、参数名和返回值必须能读懂。
5. **测试题**：写 2 个正常测试和 1 个边界测试，确认修改后行为不变。
6. **算法题**：完成 [Valid Palindrome](https://leetcode.com/problems/valid-palindrome/)，题型：双指针。提示：从两端收缩，训练边界移动和字符过滤。
7. **项目题**：把今天结果接入 `AI Builder 作品集` 的一个小模块，并在 README 里写 3 行使用说明。

### 难度控制

Medium/Hard 项目化训练，允许拆成两天，但必须有设计、测试和复盘。 每道题都要沉淀到作品集：问题定义、核心函数、测试、README、演示输出。 如果算法题暂时写不出完整代码，先写暴力思路、输入输出样例和复杂度，再补优化版。
