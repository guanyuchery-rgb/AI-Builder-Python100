## 工程问题解决结构

### 1. 是什么（What）

学习记录 CLI 是一个在终端运行的小工具。最简定义：它接收一条学习记录，把它整理成结构化数据，然后保存到本地文件。

### 2. 为什么存在（Why）

如果没有这个小工具，学习过程就只剩零散截图、终端输出和脑子里的印象。CLI 存在的意义是把“我今天学了什么”变成可保存、可复查、可继续分析的数据。

### 3. 真实工程场景（Real Scenario）

- 科研场景：每天记录读了哪篇论文、核心问题、实验想法和未解决问题。
- 数据分析场景：记录每次数据清洗的输入文件、处理步骤、输出文件和异常。
- Quant 场景：记录每天看的指标、策略假设、回测结果和参数变化。
- LLM/Agent 场景：记录 Agent 每次调用工具的输入、输出、失败原因和人工审查结论。

### 4. 怎么做（How）

先看任务流程，再看代码：

```text
输入学习主题和用时
  ↓
检查输入是否为空、用时是否合理
  ↓
整理成一条 record
  ↓
追加保存到文件
  ↓
打印保存成功的位置
  ↓
下次可以继续读取和复盘
```

```python
from pathlib import Path

LOG_FILE = Path("learning_log.csv")

def build_record(topic, minutes):
    if not topic:
        return None
    if minutes <= 0:
        return None
    return f"{topic},{minutes}\n"

def save_record(record):
    with LOG_FILE.open("a", encoding="utf-8") as f:
        f.write(record)

def main():
    topic = "Python Day10"
    minutes = 120
    record = build_record(topic, minutes)
    if record is None:
        print("invalid record")
        return
    save_record(record)
    print(f"saved to {LOG_FILE}")

if __name__ == "__main__":
    main()
```

### 5. 工程设计思维（Design）

工程师不会把输入、校验、保存全部堆在一起，因为这样后面很难改。`build_record()` 负责把输入变成记录，`save_record()` 负责保存文件，`main()` 负责串流程；多这一层拆分，是为了以后能加 JSON、SQLite、统计报告、测试或 Agent 调用，而不用重写全部代码。

### 基础知识深讲：小练习闭环与状态保存

Day10 的重点是把零散语法连成一个可用系统。一个小练习至少要有输入、处理、输出和保存。比如学习记录 CLI：输入主题和用时，处理成结构化记录，输出确认信息，并保存到文件。这样它才不是临时练习，而是一个可复查工具。

主题里的“状态”指系统当前保存了什么信息。学习记录保存到文件后，下一次运行还能读取，这就是最简单的状态管理。以后数据库、缓存、Agent memory、回测结果表，本质上都是更复杂的状态保存。

小练习不要一开始追求功能多。菜单、界面、统计图、数据库都可以以后加。最小闭环应该先保证一条记录能进入系统、被保存、被读取、被解释。这个顺序能避免你被复杂功能拖住。

### 手写顺序

- 先定义一条记录包含哪些字段，例如日期、主题、用时、收获。
- 再写最小功能：新增记录、列出记录、保存到文件。
- 最后补错误处理：空输入、文件不存在、格式不对。

### 今天必须真正弄懂

- **输入**：今天这段代码从哪里拿到数据，是手写值、终端输入、文件，还是函数参数。
- **处理**：每一行代码改变了什么状态，例如变量变了、列表多了一个元素、判断走了某个分支。
- **输出**：结果给谁看，是终端输出、返回值、文件，还是后面函数继续使用。

### 执行过程追踪

学习今天内容时，每写 5 到 10 行代码，就问自己四个问题：

Day10 的重点不是把 CLI 做大，而是学会给小练习画边界。一个最小练习至少要回答：

```python
def is_valid_topic(topic):
    return topic.strip() != ""

def build_record(topic, minutes):
    if not is_valid_topic(topic):
        return None
    if minutes <= 0:
        return None
    return {"topic": topic, "minutes": minutes}
```

#### 3. 真实主题里怎么用

- **数据分析**：先明确字段、类型、缺失值、样本范围，再写处理逻辑。不要直接对未知数据写复杂代码。

- **LLM/Agent**：所有输入输出都要有结构，模型结果要校验，高风险动作要有人类审查点。

#### 4. 常见坑和边界

- 用 `while True` 做交互输入时必须有 `break` 退出条件。
- 保存记录时优先追加，不要无意覆盖旧日志。
- 输入校验要先于保存动作。
- 不要把所有逻辑堆进一个文件或一个函数。先拆出最小核心函数，再决定是否需要 CLI、API、看板或测试。
- 不要用“能跑一次”代表掌握。真正掌握是换一份输入、换一个场景、换一个主题后仍然能迁移。
