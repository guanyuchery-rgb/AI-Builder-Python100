## 知识点结构

### 定义

把文件、CLI、日志、测试串成可交付脚本，是把一个具体问题拆成可运行、可检查、可复用代码的过程。

### 为什么存在

它让学习从临时笔记变成长期资产。公开仓库里的每一天都应该能让读者看到：今天学什么、为什么学、怎么运行、如何检查结果。

### 最小案例

```python
def summarize_minutes(rows):
    total = sum(row["minutes"] for row in rows)
    return {"count": len(rows), "total_minutes": total}

rows = [{"topic": "python", "minutes": 120}, {"topic": "sql", "minutes": 90}]
print(summarize_minutes(rows))
```

## 基础详细讲解

### 工程化小项目 DataCLI

Day20 是第一阶段收束：把文件、函数、参数、日志、测试和 README 合成一个可运行小工具。完成标准不是功能多，而是别人能看懂怎么运行，出错后你能定位。

### 手写顺序

- 先写 README 的运行命令，再按命令去实现代码。
- 保留一份小样例数据，确保任何时候都能复现结果。
- 把常见失败写进 Debug 记录，例如路径不对、格式不对、空数据。

### 今天必须真正弄懂

- **输入**：今天这段代码从哪里拿到数据，是手写值、终端输入、文件，还是函数参数。
- **处理**：每一行代码改变了什么状态，例如变量变了、列表多了一个元素、判断走了某个分支。
- **输出**：结果给谁看，是终端输出、返回值、文件，还是后面函数继续使用。

### 和后面项目的关系

这个 DataCLI 是后面数据分析管道、Quant 研究 starter 和 LLM 报告助手的共同底座。
