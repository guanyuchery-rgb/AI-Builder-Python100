## 工程问题解决结构

### 1. 是什么（What）

工程化小项目：Data CLI 是用 Python 解决工程任务时的一组小工具。最简定义：它帮助你把「小项目闭环、README、测试」放进可运行流程里。先把它当成解决问题的方法，不要先背术语。

### 2. 为什么存在（Why）

如果没有 工程化小项目：Data CLI，代码通常会变成一次性脚本：逻辑散、结果难复查、报错难定位、以后项目里也难复用。它存在的意义是把真实任务拆成更清楚的步骤，让你知道哪里输入、哪里处理、哪里输出。

### 3. 真实工程场景（Real Scenario）

- 科研场景：整理实验记录、批量生成学习日志、清洗论文标题或作者字段。
- 数据分析场景：处理一小份 CSV/JSON/文本记录，输出可检查结果。
- Quant 场景：读取几条价格或收益样例，计算简单指标或生成检查日志。
- LLM/Agent 场景：给未来 Agent tool 准备稳定的输入、输出、错误信息和本地文件。

## 基础详细讲解

### 如果你完全看不懂，先只抓这一句话

今天学的是：**把前面学过的语法装成一个别人能运行的小工具。**

不是做大项目，而是做清楚这个闭环：

```text
样例输入 -> 核心函数 -> 输出结果 -> README 说明怎么运行 -> 测试确认没坏
```

### 一个可交付小工具至少有什么

| 部分 | 作用 |
| --- | --- |
| `README.md` | 告诉别人怎么运行 |
| `src/` 或 `.py` 文件 | 放核心代码 |
| `data/` | 放样例输入 |
| `tests/` | 放测试 |
| 输出文件 | 保存结果，方便复查 |

### 为什么 README 先写

README 先写，不是形式主义。它会逼你说清：

```text
这个工具解决什么问题？
输入文件是什么？
运行命令是什么？
输出在哪里？
失败时看哪里？
```

如果这些说不清，代码通常也会写乱。

### 最小 DataCLI 数据流

```text
读取 records.json
-> 解析成 list[dict]
-> 计算总学习分钟数
-> 输出 summary
-> 保存 report.md
```

### 项目卡住怎么退回最小闭环

只保留四件事：

1. 一份样例输入。
2. 一个核心函数。
3. 一个运行命令。
4. 一个可检查输出。

其他功能都先不要加。

### 工程化小项目 DataCLI

Day20 是第一阶段收束：把文件、函数、参数、日志、测试和 README 合成一个可运行小工具。完成标准不是功能多，而是别人能看懂怎么运行，出错后你能定位。

### 基础知识深讲：从语法到可交付小工具

Day20 的深层目标是把基础语法组织成一个可交付单元。可交付不等于复杂，而是别人能运行、能理解、能复现、能定位失败。一个小工具至少包含核心函数、入口命令、样例输入、样例输出、测试和 README。

项目结构是为了降低认知负担。`src` 放代码，`tests` 放测试，`data` 放样例数据，README 放运行说明。结构清楚后，六个月后重新打开项目，你不用重新猜每个文件的作用。

可复现是工程学习和 Quant 学习的共同底线。如果结果只在你当前电脑、当前目录、当前临时数据下能跑，它还不是可靠工具。可复现要求运行命令固定、样例数据固定、输出解释固定、依赖说明固定。

Day20 之后你会进入数据分析、SQL、API、LLM、Agent、Quant。它们看起来领域不同，但底层工程要求一致：输入边界清楚，核心逻辑可测试，输出结果可复查，失败路径可定位。

### 手写顺序

- 先写 README 的运行命令，再按命令去实现代码。
- 保留一份小样例数据，确保任何时候都能复现结果。
- 把常见失败写进 Debug 记录，例如路径不对、格式不对、空数据。

### 今天必须真正弄懂

- **输入**：今天这段代码从哪里拿到数据，是手写值、终端输入、文件，还是函数参数。
- **处理**：每一行代码改变了什么状态，例如变量变了、列表多了一个元素、判断走了某个分支。
- **输出**：结果给谁看，是终端输出、返回值、文件，还是后面函数继续使用。

这个 DataCLI 是后面数据分析管道、Quant 研究 starter 和 LLM 报告助手的共同底座。

## 工程工作流补充

### 如果你完全看不懂，先只抓这一句话

今天这章不是孤立知识点，而是在训练一个真实工程习惯：把「工程化小项目DataCLI」放进一个可以运行、可以检查、可以复用的小流程里。

这一阶段的目标是把基础语法连接到文件、异常、模块和小项目。你要开始习惯“代码会长期运行、别人会复用、以后自己会回来维护”。

### 真实工作流怎么想

你可以按下面这条线理解今天内容：

```text
任务目标 -> 拆成函数 -> 处理异常 -> 保存结果 -> 写 README/复盘
```

这条线的意思是：先不要追求写得高级。先保证你知道数据从哪里来，经过了哪些变量，最后去了哪里。只要能画出这条数据流，你就已经从“背写法”进入“解决问题”。

### 执行过程追踪

学习今天内容时，每写 5 到 10 行代码，就问自己四个问题：

1. 现在有哪些变量？
2. 每个变量是什么类型？
3. 下一行代码会改变哪个变量？
4. 如果输出不对，我应该打印哪里？

这四个问题比多背几个函数更重要。以后做数据分析、Agent tool、量化回测时，真正救命的就是这种追踪能力。

### 最容易混的点

- 不要只看最终答案，要看中间变量怎么变化。
- 不要把函数、循环、文件、列表当成分开的知识，它们最后都会组合在一个任务里。
- 不要一开始就追求“优雅写法”。先写清楚，再减少重复。
- 每个例子都要能说清楚 input、process、output。

### 今日最小工程化练习

在今天主代码旁边新建一个 `day20_mini_project.py`，只做三件事：

1. 准备一小份样例数据。
2. 用今天知识点处理它。
