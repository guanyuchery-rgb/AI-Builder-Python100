## 基础详细讲解

### 如果你完全看不懂，先只抓这一句话

今天学的是：**一个项目不要把所有代码堆在一个文件里，稳定逻辑放到模块里，运行入口放到 main 里。**

```text
utils.py  -> 放可复用函数
main.py   -> 放运行入口
```

模块就是 `.py` 文件。你从另一个文件里拿函数用，就是 `import`。

### import 到底在做什么

假设 `utils.py` 里有：

```python
def add(a, b):
    return a + b
```

`main.py` 里写：

```python
from utils import add

result = add(2, 3)
print(result)
```

执行含义：

| 代码 | 发生了什么 |
| --- | --- |
| `from utils import add` | 从 `utils.py` 里拿出 `add` 函数 |
| `add(2, 3)` | 调用这个函数 |
| `print(result)` | 输出结果 |

### `if __name__ == "__main__"` 先这样理解

```python
def main():
    print("run")

if __name__ == "__main__":
    main()
```

这句话的意思是：**只有直接运行这个文件时，才执行 main；如果这个文件被别人 import，就不要自动执行。**

### 最容易错的地方

| 现象 | 原因 | 怎么改 |
| --- | --- | --- |
| `ModuleNotFoundError` | Python 找不到那个 `.py` 文件 | 确认两个文件在同一目录，或从项目根目录运行 |
| import 时自动打印一堆东西 | 被导入文件里有直接执行代码 | 把执行代码放进 `main()` 和入口保护 |
| 文件名和库名冲突 | 例如文件叫 `json.py` | 改成更具体的名字 |

### 模块、包与可复用脚本

模块就是一个 `.py` 文件，包是一组模块。把代码拆成模块，是为了让项目从“一个长脚本”变成“多个职责清楚的小文件”。`import` 不是炫技，而是复用。

### 基础知识深讲：模块边界与导入机制

一个 `.py` 文件就是一个模块。模块化的目的不是把代码拆碎，而是给不同职责划边界。比如 `metrics.py` 放指标计算，`main.py` 放运行入口，`utils.py` 放通用辅助函数。文件名就是模块名，所以命名要短、清楚、可导入。

`import` 的意思是让当前文件使用另一个模块里的名字。导入时 Python 会按搜索路径寻找模块，最常见的是当前目录和环境里的包。如果导入失败，先检查文件位置、文件名、运行目录，而不是马上怀疑语法。

`if __name__ == "__main__":` 是 Python 项目里非常重要的入口保护。一个文件被直接运行时，`__name__` 是 `"__main__"`；被别的文件导入时，`__name__` 是模块名。这样可以避免导入函数时顺便执行一堆脚本逻辑。

模块边界会直接影响项目维护。后面你做数据工具、FastAPI、Streamlit、Agent tool 时，如果核心函数和运行入口混在一起，测试和复用都会困难。越早建立入口和逻辑分离，后面越省心。

### 手写顺序

- 把计算函数放进 `utils.py`，把运行入口放进 `main.py`。
- 用 `if __name__ == "__main__":` 区分被导入和直接运行。
- 每个文件只承担一个主要职责，先少拆，再逐步拆。

### 今天必须真正弄懂

- **输入**：今天这段代码从哪里拿到数据，是手写值、终端输入、文件，还是函数参数。
- **处理**：每一行代码改变了什么状态，例如变量变了、列表多了一个元素、判断走了某个分支。
- **输出**：结果给谁看，是终端输出、返回值、文件，还是后面函数继续使用。

### 和后面项目的关系

后面 FastAPI、Streamlit、Quant backtest、Agent tool 都会依赖清晰模块结构。

### 零基础拆解：模块拆分和 main 入口

#### 这一天到底在练什么

模块拆分是为了让代码容易找。一个文件放工具函数，一个文件负责运行入口。这样项目变大时不会全部挤在一个脚本里。

#### 手写时按这个顺序

- 先写 `utils.py`，放一个简单函数。
- 再写 `main.py`，从 utils 导入函数。
- 最后加 `if __name__ == "__main__":`，让入口清楚。

#### 最常见卡点

导入失败时，先确认两个文件是否在同一目录、文件名是否拼对、是否从正确目录运行。

#### 自查问题

- 我能解释 `import` 是复用另一个文件。
- 我能把函数和运行入口分开。
- 我能从终端运行 main 文件。

## 工程化详细讲解

### 如果你完全看不懂，先只抓这一句话

今天这章不是孤立知识点，而是在训练一个真实工程习惯：把「模块、包与可复用脚本」放进一个可以运行、可以检查、可以复用的小流程里。

这一阶段的目标是把基础语法变成可复用的小能力。你要从“我会写一段代码”，过渡到“我知道这段代码放在哪里、输入是什么、输出是什么、失败时怎么查”。

### 真实工作流怎么想

你可以按下面这条线理解今天内容：

```text
输入数据 -> 选择数据结构 -> 写成函数 -> 打印/保存结果 -> 记录错误
```

这条线的意思是：先不要追求写得高级。先保证你知道数据从哪里来，经过了哪些变量，最后去了哪里。只要能画出这条数据流，你就已经从“背语法”进入“写程序”。

### 执行过程追踪

学习今天内容时，每写 5 到 10 行代码，就问自己四个问题：

1. 现在有哪些变量？
2. 每个变量是什么类型？
3. 下一行代码会改变哪个变量？
4. 如果输出不对，我应该打印哪里？

这四个问题比多背几个函数更重要。以后做数据分析、Agent tool、量化回测时，真正救命的就是这种追踪能力。

### 最容易混的点

- 不要只看最终答案，要看中间变量怎么变化。
- 不要把函数、循环、文件、列表当成分开的知识，它们最后都会组合在一个任务里。
- 不要一开始就追求“优雅写法”。先写清楚，再减少重复。
- 每个例子都要能说清楚 input、process、output。

## 5 道基础巩固题

#### 1. 消息函数

题目：写函数返回学习消息。

讲解：模块里先放小函数。

```python
def build_message(topic):
    return f"今天学习：{topic}"

print(build_message("module"))
```

#### 2. main 函数

题目：用 main 管理入口。

讲解：项目脚本常见结构。

```python
def main():
    print("run main")

main()
```

#### 3. 入口保护

题目：添加 `__main__` 判断。

讲解：方便未来被 import。

```python
def main():
    print("safe run")

if __name__ == "__main__":
    main()
```

#### 4. 拆分处理逻辑

题目：一个函数做格式化，一个函数做输出。

讲解：避免所有逻辑混在一起。

```python
def format_topic(topic):
    return topic.strip().title()

print(format_topic(" python "))
```

#### 5. 写模块说明

题目：用注释说明模块用途。

讲解：开源代码要能被别人读懂。

```python
# day09_module.py
# 作用：演示 main 入口
print("ok")
```
