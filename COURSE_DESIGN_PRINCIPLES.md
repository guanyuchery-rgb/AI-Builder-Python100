# AI Builder Python100 学习内容重构原则

本课程不是语法学习路线，而是工程问题解决路线。

更准确地说：

本课程不是承诺某几个主题一定有用，而是训练“如何做工程”的过程。

工程名只是训练场景。

真正要掌握的是：

- 如何发现小问题。
- 如何定义输入输出。
- 如何做最小版本。
- 如何处理失败。
- 如何写 README、测试、日志和复盘。
- 如何判断工程是否值得继续。

核心顺序从：

```text
语法 -> 定义 -> 例题
```

改为：

```text
问题 -> 场景 -> 为什么存在 -> 怎么解决 -> 语法实现
```

## 新库 / 新 API 首次出现规则

从 Day11 开始，凡是第一次引入新库、新对象、新函数或新 API，必须先给学习者一张认知地图。

不要直接写复杂代码。

先回答：

```text
这个库是干什么的？
它里面有哪些核心工具？
每个工具的输入是什么？
每个工具的输出是什么？
这个工具和前面学过的知识有什么关系？
```

例如引入 `csv` 时，必须先写：

| 工具 | 输入 | 输出 | 解决的问题 |
| --- | --- | --- | --- |
| `csv.reader` | CSV 文件 | 一行行 list | 读取没有字段名意识的表格 |
| `csv.writer` | list / tuple | CSV 文件 | 把列表形式的数据写成表格 |
| `csv.DictReader` | CSV 文件 | 一行行 dict | 用字段名读取表格 |
| `csv.DictWriter` | dict | CSV 文件 | 用字段名写入表格 |

然后再讲代码。

第一次出现类似下面的代码时：

```python
writer = csv.DictWriter(
    f,
    fieldnames=["day", "topic", "minutes"],
)
```

必须逐项拆解：

- `csv` 是什么。
- `DictWriter` 是什么。
- `f` 是什么。
- `fieldnames` 是什么。
- 返回的 `writer` 能做什么。

这一条同样适用于后续的 `json`、`re`、`datetime`、`pandas`、`numpy`、`sqlite3`、`requests`、`FastAPI`、`Streamlit`、`sklearn`、`torch`、`transformers`、`pydantic`、LLM/Agent SDK、Quant 相关库。

机器学习、深度学习、Transformer 和 PyTorch 放在 Day 主线里时，只承担基础认知职责：

- 这个库解决什么问题。
- 核心对象是什么。
- 输入和输出是什么。
- 和 LLM / Agent 的关系是什么。
- 最小代码如何运行。

不在 Day 主线里训练大模型，不把完整 AI 系统塞进单日课程。

教学顺序固定为：

```text
认知地图
  ↓
是什么
  ↓
为什么存在
  ↓
不用它会怎样
  ↓
和旧知识的连接
  ↓
最小语法
  ↓
逐参数拆解
  ↓
最小代码
```

## 每个知识点统一结构

Day01-Day100 的主线职责是知识学习，不承担大型工程任务。

每个 Day 的学习量上限是 **4 小时以内**。

推荐节奏是 2 小时左右。

如果某一天的内容需要超过 4 小时才能完成，说明它已经不是 Day 主线内容，必须拆出去：

- 大型工程任务放到 `工业化挑战/`。
- 作品沉淀放到 IC。
- Day 只保留知识、原理、API、小型示例和 Debug。

统一模板：

```text
What
Why
How
Common Errors
Future Usage
```

## 加厚理解规则

加厚不是重复解释，也不是增加大型项目。

每个 Day 的加厚只允许做 8 层：

```text
认知地图
  ↓
输入 -> 处理 -> 输出
  ↓
最小可运行代码
  ↓
逐对象拆解
  ↓
错误实验
  ↓
旧知识连接
  ↓
迁移场景
  ↓
IC100 衔接
```

具体执行规则见：[PYTHON100_THICKENING_GUIDE.md](PYTHON100_THICKENING_GUIDE.md)。

这不是删除已有 Day 内容，而是调整使用方式：

- Day 保留为知识主线。
- IC 作为能力验证体系。
- 已有 Day 中的工程名保留为场景或旧版说明。
- 后续新增内容优先遵守 Day / IC 解耦。

### 1. 是什么（What）

用不超过 3 句话解释概念，避免复杂术语，并给出一句最简定义。

### 2. 为什么存在（Why）

回答：如果没有这个东西，会出现什么问题。

### 3. 真实工程场景（Real Scenario）

语法讲解之前，先说明它会在哪些真实任务中出现：

- 科研场景
- 数据分析场景
- Quant 场景
- LLM/Agent 场景

### 4. 怎么做（How）

先给流程，再给代码：

```text
场景
  ↓
流程图
  ↓
代码
```

### 5. 工程设计思维（Design）

解释工程师为什么这样写，为什么不直接写，为什么要多这一层。

### 6. 工程中的位置（Where）

说明这个知识点未来会在哪些工程中再次出现，例如：

- Quant 回测框架
- FastAPI / Streamlit
- Agent tool
- CLI 工具
- 数据处理脚本
- GitHub 开源作品


## 课程重构原则

禁止把新内容写成：

```text
定义 -> 语法 -> 例题
```

统一改为：

```text
真实任务
  ↓
拆解流程
  ↓
引出问题
  ↓
引出知识点
  ↓
解释设计原因
  ↓
代码实现
  ↓
未来工程应用
```

## 掌握标准

课程目标不是“学会 Python 语法”，而是学会把真实任务拆成函数、模块、流程和工程结构。

每个知识点最终必须能回答四个问题：

1. 它是什么？
2. 它为什么存在？
3. 它解决什么问题？
4. 它在未来 Quant / LLM / Agent 作品中怎么使用？

只有能够回答这四个问题，才算真正掌握。

## 实现过程化原则

从 Day10 开始，所有所谓“工程”都要降级为过程练习。

写作时不要默认说：

```text
今天我们做一个有用工程。
```

而要说：

```text
今天我们练习一个做工程的过程。
```

统一过程：

```text
模糊想法
  ↓
真实问题
  ↓
使用者
  ↓
输入
  ↓
输出
  ↓
最小版本
  ↓
失败点
  ↓
代码实现
  ↓
交付材料
  ↓
复盘和下一步
```

## 聊天式学习风格

所有正文逐步改成更接近 GPT 对话的风格。

要求：

- 单段短。
- 一段只讲一个意思。
- 先说人话，再说术语。
- 先问问题，再给代码。
- 不要一口气倒完整百科。
- 总体内容可以长，但每次阅读要轻。
- 每个小节最后给一个很小的下一步。

核心原则：

```text
单次短，总体长。
```

不是减少字数。

是降低每次理解的压力。
